Use with Git and Subversion

Source control system is an absolute necessity in modern software development. Depending on type of the project you might want to use either Git or Subversion. Agile Toolkit uses Git due to it's open-source nature. If you are willing to add changes to Agile Toolkit, you can fork it on GitHub.com.

Git introduction

If you are willing to use most up-to-date copy of Agile Toolkit, then you would need to use "git". Git is a command-line utility which is executed as "git command arguments ..'. A good starting guide on Git is: http://book.git-scm.com.

https://github.com/atk4/atk4 is GitHub page for Agile Toolkit. You will find "branch" switch as well as selection for tags. There are two major branches: "devel" and "master". All the stable releases are based on "master" branch and beta releases are based on "devel" branch. Sometimes new feature branches will appear as experimental features are being developed.

Documentation on this site applies to 4.1 branch (or later where specified). Throughout the code improvements some functions might be renamed. A backwards compatibility is maintained throughout one major release, for instance migrating 4.0 code to 4.1 might work well, but migrating to 4.2 might require you to change calls to some obsolete functions. Some older classes may be removed and placed under "atk4-obsolete" repository. If your code relies on them, copy them into local lib or use addLocation.

    • Share your project

    If you are developing non-commercial project with Agile Toolkit under AGPL license, we invite you to share this project on GitHub.

    • Open github.com, login and create new repository. Follow instruction.
    • Remove atk4 / atk4-addons if you have downloaded them through ZIP
    • Checkout atk4 and atk4-addons directly through Git
      $ git submodule add git@github.com:atk4/atk4.git $ git submodule add git@github.com:atk4/atk4-addons.git $ git submodule update
    • If you need development branch:
      $ cd atk4 $ git checkout devel $ cd .. $ cd atk4-addons $ git checkout devel $ cd ..
    • use "git status" and create your own .gitignore files. Add "config.php", ".htaccess" and other files which you do not wish to share in there. Why placing config.php and .htaccess into Git / SVN is bad?

      Config file contains server-specific configuration. It might be private keys, or mysql password or some other sensetive information. Adding this information into Git may expose it on-line or to people who woludn't normally have access to the server.

      File ".htaccess" might be OK for adding, although if you have some server-specific configuration in there, the same principle may apply.

    • "git add ." to add all files to your new repository
    • "git push" to send your files to Github.com.
