Object Oriented Web Programming

OOP is one of the major concepts in the world of Computing. For decades Object Oriented Programming was used in all environments.

Web Programming, however, was initially design-driven. Scripting was supposed only to fill the gaps in static HTML files.

Agile Toolkit introduces the "Real" object-oriented web development. The principle of "abstraction" and "black boxes" is applied on all parts of web development - Models, Views, Controllers, Pages and Applications itself.

What are Web Objects?

$page->add('Button')->set('This is object');

Observe the button on the right. To your browser it's a sequence of HTML code. However for developer in Agile Toolkit, the button is the object. Developer may not want to know, how button presents itself, how it reacts to events or how it's communicating with the database. Like many other things around it — button is yet another object.

Without thinking too much about "how?" a developer can now focus on "what?". Need More info about Object-Oriented Programming?

Objects is our real life. Take a car for example. Car is an object. It has properties such as color, model, year. There are things you can do with a car - drive, wash, lock and unlock.

The goal of Object Oriented Programming is to discover a self-sufficient objects in the software and describe them using classes. Behaviour and properties of the button are described in the Button's class.

Properly constructed Object-oriented architecture hides the complexity of things while still allowing a developer to be in control of the behaviour. With the great Object-oriented capabilities since PHP5 web applications can also become very object-oriented.

Agile Toolkit implements the general architecture as well as large amount of base classes. You can use the classes you like and build upon them. Programming with Agile Toolkit is like building a sky-scraper. You re-use, extend, enhance. You build upon the foundation, not merely use it as a code-generator.

Object-Oriented universe of Agile Toolkit

When Object Oriented Design is applied to Web Development it abstracts the way how you operate with Web Objects such as forms, buttons, pages, links and other. Instead of writing HTML, CSS ad JavaScript code you now interact with a native object in PHP.

The button example demonstrates just one object in Agile Toolkit. A more complex objects can implement forms, signup wizards, payment forms, file manager, polls, comment feed an so on. Regardless of which object you are using, it's straightforward and simple.

When developing with Agile Toolkit, you will design your objects too. Primarily you would need to deal with two types of objects: Models and Pages. However, there are five major object types in Agile Toolkit.

Fives types of objects in Agile Toolkit

I have only mentioned "Visual" objects or views. There are actually 5 objects types in Agile Toolkit and they are described below.

Application (API)

In simple terms, web application helps user manage server-side data. For each request by the user, web application needs to collect information from the data storage and produce response to that user.

Pages

Page is a page. That's what we see when we open a book. It's also what we see when we open a website. On a websate we navigate between the pages. Pages can be different, but can also share common components such as look or layout.

It makes sense to build page through object oriented approach. This way one page may inherit another. For example you might have a generic signup page which is further extended by similar registration form for different scenarios, like in our example.

In Agile Toolkit — Pages define UI logic. Often skilled designer could understand Page definitions or even change some of the PHP code. It's essential to separate UI logic from business logic, which is defined in Models.

Models

Models are objects mimicking a real-world objects. Depending on your application, models will be defined differently. For instance for job site you may have JobApplication, User and similar other objects.

Again, object-oriented approach makes a lot of sense for modeling your application classes. If you have different user types, they would extend same parent class.

Some plug-ins may include some generic models. For instance there is model for "Country" along with database table with content. It contains official list of ISO countries and codes. Including such a model can save you development time and help you update your application when country list is updated or changed.

Views

Views are Widgets. Except that the View is also implemented in Object-Oriented design - as a class. One View can inherit another View or it can contains some other views within itself.

Unlike widgets which are oriented at end-users, you, as developer, understand benefit of object-oriented approach much better. That's why views in Agile Toolkit are not designed to serve all purposes, instead they are designed to be simply extended and tweaked the way you need.

Controllers

Your application needs a lot of functionality. Some would help you deal with specific problems such as template parsing or database access.

Controllers in Agile Toolkit enhance functionality of their parent object. Controller may enhance the way how errors are displayed, how audit is done on your models or how pagination works in your grid.

As controllers often have common things, object-oriented architecture helps here out. For instance you could have different authorization mechanisms based on generic authorization class.

What is Abstraction?

Back to our early example of a button object. The Button object is designed to do one thing only — implement a button. Form object on other hand is meant to implement forms. It may also rely on Button to do it's job.

Going further up, a Page object's job is to implement a single page. Page relies on objects such as Forms, Buttons blocks of text and so on. Finally the top-most object, which relies on Pages, Authentication, Database connectivity and other objects and which is meant to implement your whole web application is the Application object.

All of the objects have few qualities in common. They rely on each-other to do their job. They hide complexity of it's implementation in their own interface.

As a good and efficient developer, your job is to understand the importance of abstraction and be able to implement additional objects to fit in their proper roles. Objects must always take advantages of other objects if possible and each object must be simple to use.

Moving forward
